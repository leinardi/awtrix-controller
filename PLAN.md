# PLAN.md — awtrix-controller Incremental Implementation Plan

## Context

awtrix-controller is a new Go application that embeds an MQTT broker to manage one or more
Awtrix3 LED matrix displays. It was previously scaffolded by copying the structure of
`swarm-scheduler-exporter` (a Prometheus + Docker Swarm exporter), so the current `go.mod`
references the wrong dependencies. The application has **no HTTP interface** — all communication
is via MQTT (comqtt). This plan replaces the swarm-exporter wiring with the correct awtrix
architecture, following CONVENTIONS.md where applicable and adapting where the two applications
diverge significantly.

The 11 work packages below build on each other in dependency order, each independently testable,
and each shipping tests alongside the code it introduces.

---

## Dependency Order

```
WP-01 (go.mod reset)
  └─> WP-02 (data models)
  └─> WP-03 (logger + clock)
        └─> WP-04 (config)
              └─> WP-05 (clientstate registry)
              └─> WP-06 (scheduler)
                    ├─> WP-07 (day/night mode)
                    ├─> WP-08 (energy-saving mode)
                    └─> WP-09 (notifications)
        └─> WP-10 (settings + broker)
              └─> WP-11 (main.go wiring)
```

## WP-02: Data Models (`internal/model`)

**Goal:** Define all JSON-serializable structs representing MQTT payloads. Zero non-stdlib
imports; no behavior — types only.

**Files to create:**

- `internal/model/stats.go` — `Stats` struct (18 fields; JSON tags match firmware names exactly)
- `internal/model/settings.go` — `Settings` struct; `BRI *int` and `ABRI *bool` are pointers so

- `internal/model/notification.go` — `Notification` and `CustomApp` structs
- `internal/model/misc.go` — `Indicator`, `MoodLight`, `Power`, `Sleep`, `Rtttl`,
  `DrawCommand`, `EffectSettings`
- `internal/model/model_test.go` — round-trip marshal/unmarshal; verify `BRI` absent from JSON

**Key design note:** `Settings.BRI` must be `*int` (not `int`) so that when energy saving is
inactive, it marshals to nothing — the spec explicitly says omit `BRI` when energy saving is off.

**Dependencies:** WP-01.

**Verification:** `go test ./internal/model/...`

## WP-06: Scheduler (`internal/scheduler`)

**Goal:** Generic one-shot timer scheduler with timezone awareness and an injectable timer
factory (for deterministic tests).

**Files to create:**

`internal/scheduler/scheduler.go`:

- `Job` struct: name, next fire time, action, reschedule func, timer, stopped flag, mutex
- `Scheduler` struct: slice of `*Job`, `clock.Clock`, `sync.WaitGroup`, `context.Context`,
  injectable `timerFactory func(d time.Duration, f func()) *time.Timer`
- `New(ctx context.Context, clk clock.Clock) *Scheduler` — uses `time.AfterFunc` as default factory
- `NewWithFactory(ctx, clk, factory)` — for test injection
- `Schedule(name string, fireAt time.Time, action func(), reschedule func(time.Time) time.Time) *Job`
  — computes `delay = fireAt.Sub(clk.Now())`, creates timer; on fire: calls `action()`,
  then `reschedule(fired)` for next time, rearms; exits cleanly if context canceled
- `Stop()` — stops all timers, waits for in-flight actions via `sync.WaitGroup`

`internal/scheduler/scheduler_test.go`:

- Fake timer factory that fires immediately when `d == 0` (set clock past fire time)
- Job fires exactly once then reschedules
- `Stop()` before fire → action never fires
- Context cancellation → clean exit
- Two concurrent jobs fire in correct order

**Dependencies:** WP-03.

**Verification:** `go test -race ./internal/scheduler/...`

## WP-07: Day/Night Mode (`internal/daynight`)

**Goal:** Sunrise/sunset calculation using `go-solar`, Day/Night mode state, polar-night
fallback, and scheduler integration with injectable `sunriseFunc` for testing.

**Files to create:**

`internal/daynight/daynight.go`:

- `Mode` type with `Day` / `Night` constants
- `ModeProvider` interface: `CurrentMode() Mode` (consumed by settings package)
- `Controller` struct: location, timezone, `clock.Clock`, `*scheduler.Scheduler`, current
  `Mode` (behind `sync.RWMutex`), `onChange func(Mode)` callback, injectable
  `sunriseFunc func(lat, lon float64, date time.Time) (rise, set time.Time, ok bool)`
- `New(loc, tz, clk, sched, onChange) *Controller`
- `Start() error` — determines initial mode from `clk.Now()`, schedules next transition
- `CurrentMode() Mode` — read-locked
- Polar fallback: if `sunriseFunc` returns `ok=false` for one event:
    - Polar night (no sunrise) → stay Night; schedule retry at next calendar day midnight in TZ
    - Midnight sun (no sunset) → stay Day; same retry strategy
    - Log `Warn` (not Error); no spin-retry

`internal/daynight/daynight_test.go` — all via `FakeClock` + fake scheduler factory:

- Clock at noon → initial mode `Day`
- Clock at 2 AM → initial mode `Night`
- Advance past sunset → `onChange(Night)` fires; `CurrentMode() == Night` (TC-02)
- Inject `sunriseFunc` returning `ok=false` for rise → Warn logged, stays Night,
  next retry scheduled at next calendar midnight (TC-17)
- Concurrent `CurrentMode()` reads during transition (`-race`)

**Dependencies:** WP-01, WP-03, WP-06. Uses `config.LocationConfig` from WP-04.

**Test cases covered:** TC-02, TC-17.

**Verification:** `go test -race ./internal/daynight/...`

## WP-08: Energy-Saving Mode (`internal/energysaving`)

**Goal:** Configurable nightly window (midnight-spanning supported), state tracking, and
scheduler integration.

**Files to create:**

`internal/energysaving/energysaving.go`:

- `Controller` struct: parsed start/end (`time.Duration` from midnight), timezone,
  `clock.Clock`, `*scheduler.Scheduler`, `active bool` (behind `sync.RWMutex`),
  `onChange func(bool)` callback
- `New(cfg config.EnergySavingConfig, tz, clk, sched, onChange) *Controller`
- `Start() error` — parses `HH:MM` start/end, determines initial `active` state, schedules
  next activation/deactivation
- `IsActive() bool` — read-locked
- `isInWindow(t time.Time) bool` — midnight-spanning check:

  ```
  if start < end: active = start <= tod < end
  if start >= end: active = tod >= start || tod < end
  ```

  where `tod` is time-of-day in the configured timezone
- `nextBoundary(from time.Time) (fireAt time.Time, willActivate bool)` — next start or end
  after `from` in configured timezone

`internal/energysaving/energysaving_test.go`:

- Clock at 01:59, window 02:00–06:00 → `IsActive()` false; advance to 02:00 →
  `onChange(true)` (TC-03)
- Advance to 06:00 → `onChange(false)` (TC-04)
- Midnight-spanning 23:00–05:00: at 22:59 inactive; 23:30 active; 04:59 active; 05:01 inactive (TC-09)
- TZ test: clock at 07:00 UTC = 02:00 EST, window start=02:00, TZ=America/New_York → activates (TC-21)

**Dependencies:** WP-03, WP-04, WP-06.

**Test cases covered:** TC-03, TC-04, TC-09, TC-21.

**Verification:** `go test -race ./internal/energysaving/...`

## WP-10: Settings Composer and MQTT Broker (`internal/settings`, `internal/broker`)

**Goal:** Settings payload builder (day/night theme × energy-saving profile) and the full
comqtt broker with authentication, connect/disconnect, and publish hooks.

**Files to create:**

`internal/settings/settings.go`:

- `Builder` struct: `config.ThemeConfig`, `daynight.ModeProvider`, `*energysaving.Controller`

    - Always: `CHCOL`, `CBCOL`, `WDCA`, `WDCI`, `TIME_COL`, `DATE_COL` from current mode's theme

- `Push(clientID string, srv *mqtt.Server, s *model.Settings) error` — marshals, calls
  `srv.Publish(clientID+"/settings", payload, true, 1)` (retain=true, QoS=1)

`internal/settings/settings_test.go`:

- Day + energy inactive → night colors absent from payload; `ABRI=true`; no `BRI` key (TC-01)
- Night + energy inactive → night theme colors
- Energy active → `BRI=1`, `ABRI=false` in JSON (TC-03)
- All 4 combinations verified via `json.Marshal(Build())` content

`internal/broker/hook.go` — `ControllerHook` embedding `mqtt.HookBase`:

- `OnConnectAuthenticate` — compare credentials; reject mismatches (TC-07)
- `OnConnect` — `registry.Register(cl.ID)`, log, call `pushSettings(cl.ID)` in goroutine

- `OnDisconnect` — `registry.Unregister(cl.ID)`, log (TC-16)

    - `*/stats` → unmarshal → `registry.UpdateStats`; log Warn on parse error

`internal/broker/broker.go`:

## WP-11: Entry Point and Wiring (`cmd/awtrix-controller/`)

**Goal:** Wire all packages, implement CLI flag parsing with env-var fallback, version flag,

**`run()` startup sequence (from SPEC §7):**

1. Parse flags (`--config/-c`, `--log-level/-l`, `--version/-v`)
2. Apply env-var fallback (AWTRIX_CONFIG, AWTRIX_LOG_LEVEL) for unset flags via `flag.Visit`
3. Handle `--version` → print `version commit date`, return 0
4. `logger.Init(resolvedLevel)`
5. `config.Load(configPath)` → return 1 on error
6. Load/validate timezone → Warn if absent (use system TZ), return 1 if invalid IANA name
7. Create `RealClock`, `Registry`, `Scheduler`
8. Create `DayNightController.Start()` → determines initial mode
9. Create `EnergySavingController.Start()` → determines initial active state
10. Create `SettingsBuilder`; define `pushSettingsToAll()` and `pushSettingsToOne(clientID)`
11. Create `ControllerHook` (injects `pushSettingsToOne`, `registry`)
12. Create broker → return 2 if port already in use
13. Create `BirthdayNotifier.Start()`, `NewYearNotifier.Start()`
14. `signal.NotifyContext(ctx, SIGINT, SIGTERM)`
15. `go broker.Serve(ctx, server)` + log startup info
16. `<-ctx.Done()`
17. Shutdown: `scheduler.Stop()` → `server.Close()` → log "shutdown complete" → return 0

**Flag registration pattern (both long and short via stdlib `flag`):**

flag.BoolVar(&showVersion, "version", false, "Print version and exit")
flag.BoolVar(&showVersion, "v", false, "Print version and exit (shorthand)")
```

```

## Cross-Cutting Implementation Notes

### Injectable Clock Pattern

Every time-dependent constructor takes `clk clock.Clock`. Production wiring passes
`clock.NewRealClock()`; tests pass `clock.NewFakeClock(fixedTime)`. The `go-solar` library
takes a `time.Time` argument, so `clk.Now()` is sufficient — no further mocking needed for polar
tests (use a real polar latitude + winter date).

### Settings Push Triggering

`onChange` callbacks wire `daynight.Controller` and `energysaving.Controller` to the same
`pushSettingsToAll()` closure in `run.go`, which iterates `registry.ConnectedIDs()` and calls
`settings.Push()` per client. The `OnConnect` hook calls `pushSettingsToOne(clientID)` in a
goroutine (non-blocking, with panic recovery) to avoid delaying the MQTT CONNECT response.

### Partial Settings Payload

The application sends **only** the 6–8 fields it manages (`CHCOL`, `CBCOL`, `WDCA`, `WDCI`,
`TIME_COL`, `DATE_COL`, and conditionally `BRI` + `ABRI`). All `omitempty` on `Settings` fields
ensures the JSON payload is minimal. Device settings not managed by the application are never
touched.

### Exit Code Discipline

- `0` — clean shutdown
- `1` — config error (file absent, invalid YAML, missing required field, invalid color)
- `2` — runtime startup error (port in use)
  `main()` is exactly `os.Exit(run())`. All `defer` cleanup in `run()` executes before exit.

## Conventions Adaptation Notes

The following conventions from `CONVENTIONS.md` should be adapted or skipped for
awtrix-controller, with rationale:

**1. Prometheus Metrics (§9) — Skip entirely.**
The application has no HTTP interface and exposes no metrics endpoint. The entire `metrics_ids.go`
pattern, `Configure…()` registration functions, `Reset()` + exhaustive-zero emission, and
`gauge.Delete()` on resource removal are all inapplicable. The `collector` package in the
reference project has no analogue here.

**2. HTTP Handlers and Server (§8) — Skip entirely.**
SPEC §1 states: "The application has no HTTP interface." There is no `server` package, no
`/metrics` or `/healthz` endpoint, no `http.Server` construction, and no HTTP graceful shutdown
pattern. The port management and graceful shutdown patterns are repurposed for the MQTT broker
lifecycle instead.

**3. Docker client and `containerd/errdefs` (§14) — Replace entirely.**
The reference application's `client.FromEnv` + `NegotiateAPIVersion` pattern is replaced by
comqtt's `mqtt.Server` construction. `errdefs.IsNotFound` error classification is not used;
standard `errors.Is` from stdlib is sufficient for the errors this application encounters.

**4. Internal `labels` package — Omit.**
The `labelutil` / `sanitize.go` pattern existed solely for Prometheus label sanitization.
No such sanitization is needed for MQTT topics (which follow a fixed schema from the device
firmware). The `internal/labels` package has no analogue in awtrix-controller.

**5. `collector` package structure — Replace with domain-specific packages.**
The reference application organized code into one `collector` package per metric family. For
awtrix-controller, the internal packages are organized by domain: `broker`, `clientstate`,
`daynight`, `energysaving`, `notification`, `scheduler`, `settings`. This matches the
CONVENTIONS.md principle of "each sub-package has a single, clear responsibility" — only the
specific package names differ.

**6. CLI short flags via stdlib `flag` — Adapt slightly.**
CONVENTIONS.md recommends stdlib `flag` (no cobra/pflag). The SPEC requires short forms
(`-c`, `-l`, `-v`) alongside long forms (`--config`, `--log-level`, `--version`). This is
achieved by registering **two separate flag names pointing to the same variable**
(`flag.StringVar(&v, "config", ...)` AND `flag.StringVar(&v, "c", ...)`). stdlib `flag`
supports this pattern. No external flag library is needed.

**7. Logging output format — Adapt to JSON.**
CONVENTIONS.md references a custom `plain_handler.go` for text-format logs. SPEC §9 says
"Structured logging (JSON or logfmt) is recommended so log aggregators can parse it." For
awtrix-controller, the logger should default to `slog.NewJSONHandler` (stdout) in production.
The `plain_handler.go` pattern from the reference project can be kept as an option (e.g., for
`--log-level debug` development output), but JSON should be the default. This is a minor
adaptation — the `internal/logger` package interface (`Init`, `L()`) stays the same.

**8. `go.mod` Go version — Use 1.22 or later.**
The reference `go.mod` lists `go 1.25.2`, which is a future/unreleased version (as of the
time of this plan). Use the latest stable release (1.22 or 1.23) that is available. The
existing Dockerfile uses `golang:1.26.0-alpine` — update to match the actual latest stable
Alpine-based Go image when implementing WP-01.
